Chapter 1 — Ingestion

5 Sources: What We Ingested and Why Each Matters

Before a single metric reaches a dashboard, the data needs to exist somewhere trustworthy. Here is what was ingested, from where, and what business question each source answers.

Most analytics platforms suffer from the same problem: no single source of truth. Google Ads data lives in one export, CM360 in another, and YouTube in a third — each with different date formats, different cost units, and different conversion definitions. The Gold fact table exists to solve this.

Why these 5 sources, not just Google Ads?

A customer might see a YouTube pre-roll, click a DV360 display ad three days later, search your brand term, convert via Google Ads — and last-click attribution gives 100% of credit to Search. The only way to correct this is to hold all channels in a single table so attribution models can compare them. That is the entire point of this pipeline.

Ingestion method: BQ Data Transfer Service

All five sources use BigQuery Data Transfer Service for scheduled daily ingestion. This keeps raw data in BigQuery natively — no intermediate cloud storage, no CSV downloads. Each transfer writes to a date-partitioned table in marketing_bronze.*. Partitioning means historical backfills are cheap: re-run a single date partition rather than the entire table.

Chapter 2 — Bronze Layer

Bronze: Ingesting Raw Data Without Losing Information

Bronze is where data lands exactly as the source sends it. No type-casting, no cleaning. This is an intentional choice — and understanding why matters for debugging any downstream issue.

The Bronze layer uses a schema-on-read approach: every column is stored as STRING. This sounds counterintuitive — why not fix dates and parse numbers at ingest? Because type-casting at ingest destroys the evidence you need when something goes wrong.

Consider the Google Ads cost_micros field. In the raw export it arrives as "338138553". If you cast this to FLOAT64 at ingest and divide by 1,000,000 in the same step, a broken division would produce R338M in spend with no way to trace it back to the source value. Keeping Bronze as STRING means you can always audit.

Chapter 3 — Apache Beam

Beam Validation: Catching Bad Data Before It Reaches the Business

31,409 rows — 6.0% of everything ingested — failed validation. This chapter explains what was checked, what failed, and why that 6.0% quarantine rate is a sign the pipeline is working correctly.

Apache Beam runs on Cloud Dataflow as a managed service. The pipeline reads Bronze, applies a ValidateAndTagRow DoFn to every row, and routes each row to one of two BigQuery sinks: Silver (valid) or quarantine (failed). Nothing is deleted — every bad row is preserved with the reason it was rejected.

Validation rules applied

  1. Date parse check

    Attempts to parse date_str against 4 format patterns. If none match, the row is tagged INVALID_DATE. YouTube was the worst offender — 3,400 rows used DD/MM/YYYY while the rest used YYYY-MM-DD.

  2. Required field presence

    8 fields must be non-NULL and non-empty. A missing campaign_id or blank impressions_str tags the row MISSING_REQUIRED. Most failures here were Google Ads rows where conversion data was absent for brand-new campaigns.

  3. Numeric range validation

    Impressions must be 0 or above. Clicks must not exceed impressions. Negative impressions in DV360 (billing adjustment rows) are quarantined as NEGATIVE_METRIC — they are real DV360 entries, not errors, but they cannot be included in ROAS without special handling.

  4. Currency code normalisation check

    Currency must be parseable as a known ISO 4217 code after trimming and uppercasing. The whitespace variant "ZAR " (trailing space) caused 1,840 Google Ads rows to fail before the trim was applied to the validation rule itself.

6.0% quarantine is healthy, not alarming

An industry benchmark for marketing data is 3–8% validation failure. At 6.0%, this pipeline is within normal range. More importantly, every quarantined row is auditable — they sit in marketing_bronze.quarantine_* with a reject_reason column and can be re-processed after the source issue is fixed. No data is ever deleted.

class ValidateAndTagRow(beam.DoFn):
    def process(self, element):
        errors = []
        date_val = self.parse_date(element.get('date_str', ''))
        if not date_val:
            errors.append('INVALID_DATE')
        for field in ['campaign_id', 'impressions_str', 'cost_micros_str']:
            if not element.get(field):
                errors.append(f'MISSING_{field.upper()}')
        if int(element.get('impressions_str', 0) or 0) < 0:
            errors.append('NEGATIVE_METRIC')
        if errors:
            element['reject_reason'] = '|'.join(errors)
            yield beam.pvalue.TaggedOutput('quarantine', element)
        else:
            yield beam.pvalue.TaggedOutput('valid', element)
Chapter 4 — Silver Layer

Silver: dbt Cleans, Types, and Deduplicates

492,071 valid rows enter Silver. Five dbt models transform them from typed strings into business-ready metrics. This is where the cost_micros problem is finally fixed — and where duplicate rows from API retries are eliminated.

dbt Core runs 5 models against the validated Silver data. Each model is incremental — only new rows are processed on each daily run. The unique_key merge strategy ensures that a row appearing twice from a Data Transfer retry replaces itself rather than doubling the metrics.

🔢
The cost_micros divide is the most common Google Ads data engineering error

A cost_micros value of 338138553 is R338.14, not R338,138,553. This single divide changes ROAS from 0.000004× to 4.2×. The dbt Silver model applies this transformation exactly once, before any ROAS is ever calculated. Every metric in the Gold layer inherits the corrected value.

-- stg_google_ads.sql (simplified)
SELECT
    PARSE_DATE('%Y-%m-%d', date_str)          AS date,
    campaign_id,
    CAST(impressions_str AS INT64)            AS impressions,
    CAST(clicks_str AS INT64)                 AS clicks,
    SAFE_CAST(cost_micros_str AS FLOAT64)
        / 1000000.0                             AS cost_zar,   -- THE CRITICAL LINE
    TRIM(UPPER(currency_code))                 AS currency_code
FROM {{ source('marketing_bronze', 'google_ads_validated') }}
WHERE _dbt_unique_key NOT IN (SELECT _dbt_unique_key FROM {{ this }})

Deduplication: the dbt unique_key on (date, campaign_id, channel_name) merges duplicate rows. 3,847 duplicate rows were removed across the analysis period — all from Data Transfer retries during a BigQuery quota incident in February 2024.

CREATE OR REPLACE TABLE marketing_gold.fct_cross_channel_performance AS
SELECT
    date,
    channel_name,
    campaign_id,
    SUM(impressions)                                              AS impressions,
    SUM(clicks)                                                    AS clicks,
    SUM(cost_zar)                                                  AS spend_zar,
    SUM(conversions)                                              AS conversions,
    SAFE_DIVIDE(SUM(clicks), SUM(impressions))                  AS ctr,
    SAFE_DIVIDE(SUM(conversions) * 89.60, SUM(cost_zar))       AS roas,
    CURRENT_TIMESTAMP()                                           AS _gold_updated_at
FROM marketing_silver.unified_paid_channels
GROUP BY 1, 2, 3;
💡
Why calculate ROAS in Gold rather than in the dashboard?

If ROAS is calculated in a BI tool, every analyst gets a slightly different number depending on how they filter dates, handle NULLs, or define the conversion window. Pre-calculating in Gold means every report, dashboard, and ad hoc query sees the same 4.2× for Google Ads. The number is auditable back to the dbt models and Beam validation that produced it.

DV360 serves 4.21M impressions — 34.8% of the total — but converts at only 0.8% per click. It receives 18.8% of budget (R278K) but delivers only 17.9% of conversions. ⚠️
DV360 diagnosis: audience targeting or creative problem?

High impressions + low CTR (1.0%) points to one of two causes: (a) wrong audience — ads are reaching people with no intent to purchase, or (b) wrong creative — the ad is not compelling enough to earn a click. To diagnose: compare DV360 audience segment performance in DV360 reporting. If top-performing segments match your GA4 purchase audiences, the problem is creative. Organic traffic is the highest-converting channel. Understanding why it out-performs paid is the key to improving paid performance — not by copying organic tactics, but by identifying where the paid journey breaks down.

  • Session to Add to cart: 12.8% — reasonable. Not every session has purchase intent. Product page quality drives this metric.
  • Add to cart to Begin checkout: 47.2% drop. Nearly half of users who add to cart never open the checkout. Common causes: price shock, forced account creation, or distraction.

💡
The checkout abandonment overlap with retargeting

Users who abandon checkout are the highest-intent segment in your data. They are very likely being retargeted via DV360 and CM360 display ads. If retargeting campaigns are not segmented by checkout abandoners specifically (using a GA4 audience exported to Display and Video 360), you are paying programmatic rates to reach cold audiences and warm ones at the same bid. Separating them and bidding up on checkout abandoners would improve DV360 ROAS without increasing total spend.

Chapter 8 — GA4 and GTM

GTM and Consent Mode: Why 24% of Your Data Is Modelled

76% of GA4 sessions have full cookie consent. The remaining 24% are modelled by Google using Consent Mode v2. This is not missing data — it is estimated data. The difference matters for how you interpret conversion numbers.

When a user declines analytics consent, GA4 receives a signal from GTM's consent mode triggers that the session exists but cannot be measured. Google's modelling fills this gap using statistical patterns from consenting users with similar characteristics. The result appears in GA4 reports as normal data — but with a modelling indicator attached.

⚠️
What 24% modelling means for this analysis

The 12,100 purchase figure includes approximately 2,900 modelled conversions. If Consent Mode modelling is directionally accurate (which it typically is for populations over 500K sessions), the headline is trustworthy. But segment-level analysis breaks down — modelled sessions cannot be attributed to specific campaigns, creatives, or keywords. ROAS is more accurate in aggregate than it is by individual campaign.

GTM consent-first tag architecture

  1. Consent Initialisation trigger fires on all pages — before any tags run
  2. OneTrust updates consent state → GTM listens via dataLayer.push
  3. GA4 Configuration tag has ads_storage and analytics_storage consent checks enabled
  4. Conversion tags (Google Ads, Floodlight) wait for ad_storage: 'granted'
  5. GA4 purchase event fires on thank-you page with ecommerce dataLayer object
✅
Consent Mode v2 is now required for Google Ads conversion import

As of March 2024, Google requires Consent Mode v2 for all European traffic using Google Ads conversion tracking. This implementation uses gtag('consent','default') with all storage types set to 'denied' before OneTrust loads, ensuring compliance. Without this, Google Ads Smart Bidding loses access to modelled conversion signals and campaign performance degrades measurably.

Chapter 9 — Data Quality

Data Quality Findings: What Failed and What It Means

94.0% of rows passed Beam validation. The 6.0% that failed are not pipeline noise — each failure category points to a fixable issue at the data source. This chapter maps each failure to its root cause and fix.

  • 1
    YouTube date format inconsistency — 3,400 rows

    Root cause: YouTube Data Transfer exports switched from YYYYMMDD to DD/MM/YYYY in January 2024 with no changelog entry. Fix: Set explicit output format parameters in the BQ Data Transfer configuration. The Beam DoFn already handles multi-format parsing — YouTube rows now pass after the format was added to the parser.

  • 2
    YouTube missing or invalid required metadata — 6,138 quarantined rows

    Root cause: A mix of Google Ads-served YouTube rows and incomplete video metadata leaves required fields blank in the transfer export. Fix: Beam quarantines invalid rows and the dbt Silver model imputes known Google Ads-served channel IDs with a channel_id_imputed = TRUE flag where attribution is still recoverable.

  • 3
    Google Ads currency trailing space — 1,840 rows

    Root cause: Google Ads exported "ZAR " with a trailing space during a 3-week window in Q1. Fix: Added TRIM(UPPER(currency_code)) to the Beam validation step. These rows now pass. The window of affected data is corrected in Silver.

  • 4
    DV360 negative impression rows — 6,802 rows

    Root cause: DV360's billing system sends adjustment rows with negative impression counts to correct over-delivery. These are legitimate DV360 behaviour, not errors. Fix: Quarantine these rows to prevent negative values entering Silver. They sit in a reconciliation table. Total DV360 impression count in Gold is net-adjusted.

Chapter 10 — Orchestration

Airflow DAG: 9 Tasks, 4 Task Groups, One Daily Pipeline

The pipeline runs automatically at 04:00 UTC daily via Apache Airflow. 9 tasks are organised into 4 logical task groups. This chapter explains the dependency structure and what each group does.

💡
Why the Gold overwrite (not append) is the right design

The Gold fact table is fully overwritten on each run. This is correct for two reasons: (a) historical ROAS changes if a delayed DV360 transfer corrects a past date, and a full overwrite ensures corrections propagate; (b) it is idempotent — running the same DAG twice produces exactly the same table. Incremental appends in the wrong order would require complex de-duplication logic in Gold that the overwrite makes unnecessary.

dag = DAG(
    dag_id='gcp_marketing_analytics_pipeline',
    schedule_interval='0 4 * * *',   # 04:00 UTC = 06:00 SAST
    start_date=datetime(2024, 1, 1),
    catchup=False,
    default_args={
        'retries': 3,
        'retry_delay': timedelta(minutes=10),
        'sla': timedelta(hours=2),    # Gold must be ready by 08:00 SAST
    }
)
Chapter 11 — Recommendations

What Should Change Next — and Why

The platform is working. Data is clean, ROAS is calculated, Airflow runs daily. These are the four highest-leverage improvements that would materially change business outcomes.

  • 1
    Add YouTube view-through attribution to the Gold model

    Why: YouTube's 2.9× ROAS is almost certainly understated — it drives brand searches that convert via Google Ads. Adding view_through_conversions from the YouTube Data Transfer export would show the true cross-channel contribution and is the single change most likely to rebalance budget allocation correctly.

  • 2
    Create a checkout-abandoner retargeting audience in DV360

    Why: 16,800 users added to cart but never purchased. Exporting this GA4 audience to DV360 and bidding 2× the current CPM floor on this segment would improve DV360 ROAS without increasing total display spend. The audience already exists in GA4 — it just needs to be activated.

  • 3
    Fix YouTube date format at source in the Data Transfer config

    Why: 3,400 quarantined rows per period from a fixable configuration issue. Correcting the export format recovers these rows without any pipeline code changes. Estimated recovery: approximately 120 attributable conversions per month currently missing from YouTube's ROAS calculation.

  • 4
    Add a consent rate monitoring alert in Airflow

    Why: Consent rate is currently 76%. If it drops below 65%, Google's Consent Mode modelling accuracy degrades significantly and ROAS figures become less reliable. An Airflow sensor task reading the GA4 consent event rate daily and alerting on a 5-point week-over-week drop would catch a CMP misconfiguration before it corrupts the Gold table.

Appendix A

SQL Reference — Key Gold Queries

A1 · ROAS by channel (all-time)

SELECT
    channel_name,
    SUM(spend_zar)                                             AS total_spend,
    SUM(conversions)                                           AS total_conversions,
    ROUND(AVG(roas), 2)                                        AS avg_roas,
    SAFE_DIVIDE(SUM(clicks), SUM(impressions)) * 100         AS avg_ctr_pct
FROM marketing_gold.fct_cross_channel_performance
WHERE date BETWEEN '2024-01-01' AND '2024-06-30'
GROUP BY 1
ORDER BY avg_roas DESC;

A2 · Weekly spend with 4-week moving average

SELECT
    DATE_TRUNC(date, WEEK)                                    AS week,
    channel_name,
    SUM(spend_zar)                                            AS weekly_spend,
    AVG(SUM(spend_zar)) OVER (
        PARTITION BY channel_name
        ORDER BY DATE_TRUNC(date, WEEK)
        ROWS BETWEEN 3 PRECEDING AND CURRENT ROW
    )                                                         AS spend_4wk_ma
FROM marketing_gold.fct_cross_channel_performance
GROUP BY 1, 2
ORDER BY 1, 2;

A3 · Budget vs conversion efficiency gap

WITH totals AS (
    SELECT channel_name,
        SUM(spend_zar) AS spend, SUM(conversions) AS convs
    FROM marketing_gold.fct_cross_channel_performance
    GROUP BY 1
)
SELECT
    channel_name,
    ROUND(spend / SUM(spend) OVER() * 100, 1)             AS budget_pct,
    ROUND(convs / SUM(convs) OVER() * 100, 1)             AS conv_pct,
    ROUND((convs/SUM(convs) OVER() - spend/SUM(spend) OVER()) * 100, 1) AS efficiency_gap
FROM totals
ORDER BY efficiency_gap DESC;
